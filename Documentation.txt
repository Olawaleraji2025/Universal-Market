Universal-Market — Layman Explanation + Mistakes & Lessons Learned

If you imagine this project as a small store website:
- People come in (Homepage)
- They browse products (Shop)
- Admin manages products and requests (Admin area)
- The website also talks to a database (Supabase)

This is what the brand does (in simple terms):
- Universal Market is an online shop that helps people buy pre-owned gadgets and appliances.
- Users can browse products, understand how ordering works, and request items (including custom orders).
- The admin area helps manage products/requests.

As I built it, I made a few common “new project” mistakes. This document explains those mistakes in plain language and what I learned from each.

1) Mistake: building without a stable route/entry structure
- What I did: I kept parts of the app commented out (admin dashboard, product pages, shop hero/product list, Supabase test component) while experimenting.
- Why it hurt: the project state became confusing; it was hard to know what was “real” and what was “temporary.”
- What I should have done instead:
  - keep one clear “baseline” that always runs (routes + layout + at least homepage + shop page)
  - move experimental code into feature flags or separate branches/components instead of commenting everything
- Lesson learned: commenting out large chunks slows you down because the repo stops being self-explanatory.

2) Mistake: leaving unfinished wiring in App.jsx
- What I did: I partially integrated components (Navbar/Footer, routes) but left other imports/usage commented and stray JSX comment blocks at the bottom of App.jsx.
- Why it hurt: App.jsx becomes messy and can confuse future-me (or any reviewer).
- What I should have done instead:
  - remove dead/commented JSX once a component is not actively used
  - create a proper route for “/admin” when ready, or isolate it behind a small wrapper component
- Lesson learned: App.jsx is the “spine” of the app—keep it clean.

3) Mistake: experimenting in-place rather than iterating safely
- What I did: I added a Supabase “tasks” component (fetch + insert) and then moved on, but I didn’t fully integrate it into the rest of the UI flow.
- Why it hurt: I learned Supabase mechanics, but the app architecture didn’t benefit from it.
- What I should have done instead:
  - after proving the query works, connect it to the actual relevant UI (e.g., product requests / admin dashboard)
  - keep the Supabase demo isolated, and create separate real components that use the same data layer
- Lesson learned: demos are good, but you still need an integration step.

4) Mistake: not standardizing data access patterns early
- What I did: I used Supabase client directly in components (fetch/insert) without a consistent abstraction.
- Why it hurt: as the project grows, logic becomes duplicated across components (loading/error/state handling).
- What I should have done instead:
  - create a small “data layer” (e.g., src/lib/supabase/* or src/services/*) with typed-ish helper functions
  - reuse common hooks for loading/error
- Lesson learned: UI code shouldn’t repeat the same data-fetch boilerplate.

5) Mistake: leaving “TODO” work unfinished without a delivery rule
- What I did: I created an Admin Dashboard component (UI-heavy) but didn’t complete the wiring (route + rendering + dev verification).
- Why it hurt: progress feels slow because you can finish UI while still being blocked by integration.
- What I should have done instead:
  - “Definition of Done” for every feature: UI + route + at least one working interaction + verified running state
- Lesson learned: always finish the integration loop.

6) Mistake: testing with placeholder/mock data instead of real contract-first design
- What I did: dashboard data like stats/requests/products currently uses mocked arrays.
- Why it hurt: UI looked good, but it wasn’t validated against real database schemas and real Supabase response shapes.
- What I should have done instead:
  - define the expected Supabase schema first (tables + column names)
  - test UI rendering with real Supabase queries as early as possible
- Lesson learned: mock data is fine for layout, but it delays discovering mismatched field names and filtering logic.

7) Mistake: not enforcing consistent component boundaries
- What I did: I ended up with many components for the homepage and shop; that’s good, but some parts (like admin/supabase tasks) were structurally different and not aligned with the same patterns.
- Why it hurt: switching between “style/patterns” makes the codebase feel inconsistent.
- What I should have done instead:
  - adopt consistent patterns: props naming, folder structure, hooks usage, and data fetching approach
- Lesson learned: consistency is performance for your future self.

8) Mistake: relying on comments for progress tracking
- What I did: I used commented blocks (e.g., large JSX comment regions) to track progress.
- Why it hurt: comments rot; they don’t enforce correctness.
- What I should have done instead:
  - use TODO.md + smaller commits + branch naming
  - keep comments for explanation, not as state
- Lesson learned: progress tracking should live in a tool (git/TODO), not in commented code.

9) Mistake: “quick fixes” that make the project harder to maintain
- What I did: e.g., in App.jsx there are stray commented sections and leftover imports/comment blocks from experiments.
- Why it hurt: it increases cognitive load and increases risk when you refactor later.
- What I should have done instead:
  - do periodic cleanup passes: remove unused code, fix formatting, ensure lint passes
- Lesson learned: refactoring is not optional—it’s part of the build.

What I’m proud of / learned positively
- I structured the UI into meaningful components (Hero, Cards, FAQ, Shop page, etc.).
- I used React Router and added a ScrollToTop helper—small UX details add up.
- I successfully wired Supabase connection and built a working fetch/insert flow.
- I created the Dashboard UI early, which made it easier to think about what admin users actually need.

“Rule” I’ll follow next time (my personal checklist)
- Keep a running baseline (never let the app become a pile of commented-out parts).
- For each feature: UI → route/wiring → real data integration → verify in dev server.
- Move data logic into shared utilities/hooks.
- Clean up commented/unused code continuously.

End result: This repo taught me that shipping isn’t just building screens—it’s integrating screens with routing, data contracts, and maintainable structure.
